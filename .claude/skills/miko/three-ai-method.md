# 3AI駆動方式

## 核心の考え方

- 実装時に余計なファイルを調査せず、docs/ を見るだけで即実装が始められる状態を常に維持する
- AIは「見る能力」はあるが「作りながら同時に見る」ができない
- 役割ごとにAIインスタンスを分離し、視点の汚染を防ぐ
- 人間と継続的に対話するのは**開発パートナー（/miko）だけ**
- 実装AI・説明AI・コードレビューAIはサブエージェント（fork）で毎回クリーン起動
- サブエージェントの試行錯誤は親に残らない。結果だけ返る
- 設計ドキュメントが「信頼の源泉」

## 3つの役割

| 役割           | 実行方式                       | 知っていること                | 知らないこと                      |
| -------------- | ------------------------------ | ----------------------------- | --------------------------------- |
| 開発パートナー | メインセッション（人間と対話） | 全ての議論経緯、docs/、otuge/ | 実装の試行錯誤                    |
| 実装AI         | サブエージェント（fork）       | design.md, plan.md            | 設計時の議論、人間との会話        |
| 説明AI         | サブエージェント（fork）       | 実装コード                    | design.md、設計意図、人間との会話 |

## スキル構成

| スキル              | 役割                                                          |
| ------------------- | ------------------------------------------------------------- |
| `/miko`             | 開発パートナー: 設計・実装の統率、簡易修正、開発フローの改善  |
| `/implement`        | 実装AI: 設計ドキュメントに基づくプラン作成・実装（fork）      |
| `/explain`          | 説明AI: 設計意図なしでコードを説明（fork）                    |
| `/design-review`    | 設計照合レビューAI: 設計とコードの照合（fork）                |
| `/code-review`      | コードレビューAI: git diff ベースの品質・セキュリティチェック |
| `/incidents-review` | incidents.md の棚卸し・ルール昇格                             |
| `/adversary`        | 反例検出AI: 議題と方針の穴を洗い出す（fork）                  |

## リポジトリ構造

```
├── .claude/incidents.md        # 障害・失敗の記録
├── docs/                       # 確定した仕様書（docs/index.md 参照）
└── otuge/                      # 実装サイクルの作業場
    └── {タスク名}/
        ├── design.md
        ├── plan.md
        ├── explanation.md
        └── research.md
```
